## 症状

開発中の対戦弾幕ゲーム（リポジトリ名: `bullet_creation`）で弾のレンダリングを実装していたとき、画面がちらつくようになりました。フレーム単位で描画内容が欠けたり戻ったりする、見ていてすぐ分かるタイプのちらつきです。

先に結論を書いておくと、最後まで残ったちらつきは**Intel Arc環境でのみ**発生し、**ドローコール発行をミューテックスで直列化したら**消えました。そして**根本原因は今も分かっていません**。この記事は原因を突き止めた話ではなく、突き止められなかった話です。

## 迷走の記録

当時のコミットログを時系列で並べると、迷走がそのまま残っています。

```
深度ステンシルバッファをD32_FLOAT_S8X24_UINTに変更
パイプラインの使用フォーマットを変更
ちらつきの原因特定、修正 原因はCPUでの並列処理の不備
スプライト描画をいろいろ改善してみた。ちらつきが出るかは知らない
使ってなかった変数を削除、リソースバリアの設定が上手くいってなかったバグを修正
ちらつき、復活。なぜなのか
とりあえず暫定的にちらつかないもの
コマンドリストへの記録はスレッドセーフのはずなのに競合するんだが？Intel許すまじ
Intelのドライバはドローコールがなぜか非スレッドセーフ
```

最初は描画パイプライン側を疑いました。深度ステンシルのフォーマットを変え、パイプラインのフォーマット設定を見直し、途中で本物のバグ（リソースバリアの設定ミス）も見つけて直しました。一度は「原因特定、修正」とまで書いています。CPU側の並列処理に不備があったという診断で、メイン環境（GTX 1660 SUPER）ではこの修正でちらつきが出なくなりました。「直った」と判断した根拠は、要するにメイン環境で再現しなくなったことでした。

そして数コミット後、「ちらつき、復活。なぜなのか」。

## 容疑者の絞り込み

再調査で分かったのは、復活したちらつきが出るのはIntel Arcを積んだ環境だけで、GTX 1660 SUPERでは同じビルドが正常に描画されるという事実です。振り返ると、最初の修正で直ったのはGTX 1660 SUPER側で再現していた症状（こちらは本当にCPU並列処理の不備だったのでしょう）で、Intel Arc側には別の問題が残っていた——のか、それとも最初から性質の違う2つの問題だったのか。ここも正直、確信はありません。

このエンジン（自作のECSエンジン上に構築しています）は描画コマンドリストの記録を複数スレッドで並列に行います。D3D12の仕様上、これは合法のはずです。1本のコマンドリストを複数スレッドで同時に触ってはいけませんが、**別々のコマンドリストを別々のスレッドで記録するのは明示的にサポートされています**。それが並列記録のためにコマンドリストとコマンドアロケータが分離されている理由のはずです。

ところが、各スレッドが自分のコマンドリストにドローコールを記録しているだけなのに、Intel Arcでだけ描画結果が壊れる。コミットメッセージの「コマンドリストへの記録はスレッドセーフのはずなのに競合するんだが？」は、このときの心境そのままです。

## 修正(と呼んでいいのか分からないもの)

最終的に入れた変更はこれだけです。

```cpp
{
    // Intel Arcドライバで起こる不具合の対処
    std::lock_guard lock(m_render_mutex);
    commandList->DrawIndexedInstanced(6, commandCount, 0, 0, 0);
}
```

スプライト描画とメッシュ描画の2箇所で、ドローコールの発行だけをミューテックスで直列化しました。これでIntel Arcでのちらつきは完全に消えました。ロックの範囲はドローコール1発の記録だけに絞ってあるので、並列記録の恩恵の大部分は残っています。

## 根本原因について正直に書く

分かっていないものは分かっていないと書くのがこの記事の趣旨なので、正直に書きます。

ドローコール記録の直列化で直るという事実から、Intelのドライバ側でコマンドリスト記録のどこかがスレッドセーフになっていないのでは、と当時のコミットメッセージには書いています。ただしこれは推測であり、確定ではありません。可能性としては少なくとも3つ残っています。

1. Intelドライバの並列記録まわりのバグ
2. 自分のコードのどこかにある未定義動作が、Intelのドライバでだけ顕在化している
3. D3D12の仕様の読み違い（「合法のはず」が実は合法ではないパターン）

2を否定しきれないのが苦しいところです。他環境では一切再現しない、という状況証拠はありますが、環境依存で顕在化するのは未定義動作の得意技でもあります。

それでもミューテックスを恒久対応として残したのは、コストがほぼゼロで、症状が確実に消え、そして対処の範囲がコメント付きの4行に閉じているからです。原因不明のワークアラウンドで一番怖いのは対処が散らばって呪いになることなので、せめて1箇所に封印して出典（この経緯）を辿れるようにしてあります。

## 教訓

- 「原因特定、修正」と書いた後に症状が復活したら、特定できていたのは原因ではなく引き金の1つだったと考える
- 環境依存の描画バグは、パイプライン設定を疑う前にまずどの環境で出るかの表を作った方が早い（今回それを最初にやらず遠回りしました）
- 根本原因に辿り着けないことはある。そのときは、ワークアラウンドの範囲を最小にして、理由をコードに書き残すところまでが仕事

## 余談

このコミットログを見返して思うのは、感情の入ったコミットメッセージは意外と資産だということです。「ちらつき、復活。なぜなのか」「Intel許すまじ」が残っていたおかげで、時系列の再構成が一瞬で終わりました。行儀の良い `fix: rendering issue` だけだったら、この記事は書けなかったと思います。
